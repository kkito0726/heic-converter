# PRD: heic-converter Webサーバーモード

CLIとして完成しているheic-converterを、Webサーバーとしても起動できるようにする。
[CLI版PRD](../cli-converter/prd.md)の「将来構想」に挙げていた `heic-converter serve` の具体化。

## 1. 背景・課題

- 現状はCLIのみで、変換したいマシンにバイナリを配置して実行する必要がある
- 社内ツール・Webアプリ・モバイルアプリなどから「HEICをアップロードして変換結果を受け取る」ユースケースに応えられない
- 設計当初からpresentation層をusecase層から分離しており(クリーンアーキテクチャ)、入力チャネルの追加はこの設計の実証でもある

## 2. ゴール

| ゴール | 内容 |
|---|---|
| マルチプロトコル | 1つのエンドポイントで **gRPC / gRPC-Web / Connect(HTTP+JSON)** の3プロトコルを受け付ける |
| REST的な使い勝手 | `curl`からJSONをPOSTするだけで変換できる(Connectプロトコル) |
| コア共有 | CLIと同一の`usecase.Converter`(デコーダ・エンコーダ)を使い、変換品質・対応形式に差が出ない |
| 単一バイナリ維持 | サーバー機能を含めても`CGO_ENABLED=0`の単一バイナリのまま |

### 非ゴール(スコープ外)

- 認証・認可(リバースプロキシ/API Gateway側の責務とする)
- レート制限・課金(同上)
- TLS終端(リバースプロキシ前提。サーバー自身はh2cで待ち受ける)
- サーバー側ローカルパスを指定した変換(パストラバーサルにつながるため受け付けない。入力は常にリクエストボディのバイト列)
- 変換結果の永続化・ジョブキュー(同期レスポンスのみ。非同期化は将来検討)

## 3. ユースケース

- **Webフロントエンド**: ブラウザからgRPC-WebまたはConnectプロトコルで直接呼び出し、iPhoneで撮った写真をアップロード前にjpgへ変換する
- **バックエンドサービス**: 他のマイクロサービスがgRPCで画像変換を委譲する
- **スクリプト/検証**: `curl`や`grpcurl`でアドホックに変換する

## 4. 機能要件

### FR-1: serveサブコマンド

```
heic-converter serve [--port 8080] [--host 0.0.0.0]
```

- 既存のCLI動作(引数にパスを渡す形)には一切影響しない
- `--port`(デフォルト8080)、`--host`(デフォルト0.0.0.0)を指定できる
- SIGINT/SIGTERMでgraceful shutdownする(処理中のリクエストは完了を待つ)

### FR-2: 変換API(ConvertService.Convert)

- リクエスト: HEIC/HEIFのバイナリ + 出力形式の配列(複数指定可) + 品質(省略時90)
- レスポンス: 形式ごとの変換済みバイナリの配列
- 1リクエスト = 1画像。複数形式を指定した場合、デコードは1回だけ行い全形式にエンコードする(CLIと同じ挙動)
- 対応形式はCLIと完全に同一(jpg / png / webp / tiff / bmp / gif)

### FR-3: 形式一覧API(ConvertService.ListFormats)

- 対応する出力形式の一覧を返す
- 副作用がないため、ConnectプロトコルではGETでも呼び出せるようにする

### FR-4: プロトコル対応

同一パス(`/heic.v1.ConvertService/...`)で以下すべてを受け付ける:

| プロトコル | 主なクライアント |
|---|---|
| Connect (HTTP/1.1 or HTTP/2 + JSON/protobuf) | curl、ブラウザfetch、connectクライアント |
| gRPC (HTTP/2) | 各言語のgRPCクライアント、grpcurl |
| gRPC-Web | ブラウザ |

### FR-5: 運用系エンドポイント

- **ヘルスチェック**: gRPC Health Checking Protocol準拠(k8s等のprobeから利用可能)
- **サーバーリフレクション**: `grpcurl`がprotoファイルなしで呼び出せるようにする

### FR-6: エラー応答

- Connectのエラーコード体系で返す(gRPCステータスコードと互換)
- 不正な形式指定・壊れた画像 → `invalid_argument`、サーバー内部エラー → `internal`
- エラーメッセージにサーバー内部のパス等の情報を含めない

## 5. 非機能要件

| 項目 | 要件 |
|---|---|
| リクエストサイズ上限 | デフォルト64MiB(フラグで変更可)。超過時は`resource_exhausted` |
| タイムアウト | クライアント指定のデッドラインを尊重しつつ、サーバー側上限も設ける |
| 並行性 | ステートレス。複数リクエストの同時処理が可能(デコーダ/エンコーダはCLIの並列変換で実績あり) |
| 配布 | 既存と同じ単一バイナリ・クロスコンパイル対応を維持 |
| 観測性 | リクエストログ(メソッド・所要時間・ステータス)を構造化ログで出力 |
| テスト | カバレッジ80%以上を維持。in-processでの3プロトコル疎通テスト |

## 6. 将来構想

- `google.api.http`アノテーション + [vanguard](https://github.com/connectrpc/vanguard-go)による純粋なREST(`POST /v1/convert`のようなリソース指向パス)への透過変換
- streaming RPCによる大容量ファイル・複数ファイルの逐次変換
- 非同期ジョブ化(アップロード → ジョブID発行 → 結果取得)

## 7. 成功基準

- [ ] `heic-converter serve`で起動し、Ctrl-Cでgraceful shutdownする
- [ ] `curl`(Connect + JSON)で実HEICファイルをjpg/pngに変換できる
- [ ] `grpcurl`(gRPC + リフレクション)で同じ変換ができる
- [ ] CLIの既存機能・テストに影響がない(全テスト成功)
- [ ] カバレッジ80%以上を維持
- [ ] `CGO_ENABLED=0`のクロスコンパイルが引き続き通る
